Linux Packages

Linux Packages: Upgrade.

Upgrade safely with pre-checks, risk controls, rollout verification, and rollback awareness.

Upgrade

Linux upgrades replace binaries and may restart systemd services. Keep the cluster healthy before upgrading nodes one at a time.

RiskWhat Can HappenMitigation
Package restartThe service may restart after package installation.Upgrade one node at a time and verify cluster health after each node.
Config incompatibilityNew binary may require a new env value or reject an old value.Re-render service env files with li9-s3-render-config before restart.
Metadata migrationMetadata format migration may be irreversible.Take metadata backup or snapshot before upgrade.
Kernel or disk changesOS patching can change disk or network behavior.Separate OS maintenance from Li9 S3 package upgrade when possible.
 
export LI9_SCHEME=http
export LI9_CONTROL_HOST=10.10.10.11
export LI9_GATEWAY_HOST=10.10.10.11
export LI9_STORAGE_NODE_URLS="${LI9_SCHEME}://10.10.10.11:9008,${LI9_SCHEME}://10.10.10.12:9008,${LI9_SCHEME}://10.10.10.13:9008"
curl -fsS "${LI9_SCHEME}://${LI9_GATEWAY_HOST}:9000/v1/info"
curl -fsS "${LI9_SCHEME}://${LI9_CONTROL_HOST}:9001/v1/info"
sudo systemctl stop li9-s3-storage.target li9-s3-control.target li9-s3-gateway.target li9-s3-storage-node.target || true

# RPM:
sudo dnf upgrade -y li9-s3-storage

# DEB:
sudo apt-get update
sudo apt-get install --only-upgrade -y li9-s3-storage

sudo /opt/li9/s3-storage/bin/li9-s3-render-config \
  --node-host 127.0.0.1 \
  --control-host "${LI9_CONTROL_HOST}" \
  --gateway-host "${LI9_GATEWAY_HOST}" \
  --storage-node-urls "${LI9_STORAGE_NODE_URLS}" \
  --write-quorum 2 \
  --region li9-mgm01 \
  --allowed-regions li9-mgm01,us-east-1
sudo systemctl start li9-s3-storage-node.target
sudo systemctl start li9-s3-control.target li9-s3-gateway.target || true
sudo systemctl status 'li9-s3@*.service' --no-pager
curl -fsS "${LI9_SCHEME}://${LI9_GATEWAY_HOST}:9000/v1/info"
curl -fsS "${LI9_SCHEME}://${LI9_CONTROL_HOST}:9001/v1/info"

Post-Upgrade Verification

After each node upgrade, verify local service health, cluster membership, metadata quorum, and an external S3 write/read/delete smoke test. Continue to the next node only after the current node is healthy and no repair or replication backlog is unexpectedly growing.

sudo journalctl -u 'li9-s3@*.service' --since "15 minutes ago" --no-pager
systemctl is-active li9-s3@storage-node.service
curl -fsS "${LI9_SCHEME}://${LI9_GATEWAY_HOST}:9000/v1/info"
curl -fsS "${LI9_SCHEME}://${LI9_CONTROL_HOST}:9001/v1/info"
aws --endpoint-url "${S3_ENDPOINT}" s3api list-buckets

Rollback Guidance

Package rollback is node-local, but runtime compatibility is cluster-wide. Do not downgrade one node below the supported cluster version if metadata WAL, erasure layout, encryption metadata, or API storage format changed. If rollback is approved, drain client traffic from the node, install the previous package, restore the previous config, and rejoin it only after validation.
