Overview Usage Services Architecture
Application integration

Use Li9 S3 like a normal S3 endpoint.

Applications do not need a Li9-specific SDK. They need an endpoint URL, region, bucket name, access key, secret key, and optionally a session token. The recommended OpenShift path is to consume these values from the Secret created for bucket access.

Consumption Workflow

1

Request bucket

Create the bucket through the operator, OBC/COSI, Helm-managed automation, or platform provisioning workflow.

2

Receive Secret

The platform delivers endpoint, bucket, region, access key, secret key, and optional session token.

3

Configure client

Use standard AWS CLI or SDK configuration with --endpoint-url or the SDK endpoint override.

4

Validate behavior

Run head, put, get, list, tag, multipart, and policy-specific tests before moving production traffic.

AWS CLI Smoke Test

Run this from a workstation or from a temporary pod that has network access to the S3 endpoint.

export AWS_ACCESS_KEY_ID=<from-secret>
export AWS_SECRET_ACCESS_KEY=<from-secret>
export AWS_SESSION_TOKEN=<optional-from-secret>
export AWS_DEFAULT_REGION=<region-from-generated-secret>
export AWS_EC2_METADATA_DISABLED=true

aws --endpoint-url https://<s3-route-or-service> s3api head-bucket --bucket app-data
aws --endpoint-url https://<s3-route-or-service> s3 cp ./payload.txt s3://app-data/payload.txt
aws --endpoint-url https://<s3-route-or-service> s3 cp s3://app-data/payload.txt ./downloaded.txt
aws --endpoint-url https://<s3-route-or-service> s3api put-object-tagging \
  --bucket app-data \
  --key payload.txt \
  --tagging 'TagSet=[{Key=env,Value=prod}]'
Expected result: head-bucket returns success, uploaded bytes round-trip without checksum mismatch, object tags are visible through S3 APIs, and unauthorized credentials receive access-denied responses instead of storage errors.

Credential Delivery

The recommended OpenShift flow is to create a bucket and access resource through the platform interface. The operator writes a Kubernetes Secret that contains the endpoint, bucket, region, access key, secret key, and optional session token.

envFrom:
  - secretRef:
      name: my-bucket-access

Configuration Values

ValueWhere it comes fromHow the client uses it
Endpoint URLRoute, service DNS name, load balancer, or native install endpoint.Passed as --endpoint-url or SDK endpoint override.
RegionPlatform default or bucket access Secret.Used for SigV4 request signing.
BucketBucket CR, OBC/COSI claim, Helm values, or platform request.Used as the S3 bucket name in API calls.
Access key and secret keyGenerated by the identity plane and stored in a Secret.Used by AWS CLI and SDK credentials provider chains.
Session tokenOptional, depending on access configuration.Used when temporary credentials are issued.

Fit Guidance

Use it when

The platform needs S3-compatible storage fully controlled inside OpenShift or on company-managed infrastructure.

Validate first when

The workload depends on strict latency targets, high multipart throughput, object lock governance, lifecycle timing, or degraded write behavior.

Do not use it when

The requirement is direct AWS-managed integrations such as EventBridge, SNS, SQS, Lambda triggers, or AWS account-level IAM semantics.

Production Validation

  • Verify AWS CLI and SDK operations against the same endpoint that applications will use.
  • Test authorized and unauthorized requests so policy failures are distinguishable from storage failures.
  • Run multipart upload, object tags, metadata, list by prefix, delete, and read-after-write checks.
  • Confirm metrics, audit records, access logs, and alert rules before onboarding production workloads.
  • For protected data modes, test one-node outage during continuous writes and verify object integrity after healing.
