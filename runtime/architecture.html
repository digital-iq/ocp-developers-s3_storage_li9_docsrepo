Runtime architecture

Independent planes with a closed failure model.

The runtime is split into narrow services so authorization, metadata, object placement, encryption, audit, and repair can be tested and released independently. The core safety rule is simple: data is committed only after the selected protection mode has durably accepted the write.

Runtime Planes

The plane model keeps responsibilities explicit. Each plane can evolve without forcing every runtime component to be rebuilt or retested from scratch.

API plane

S3 protocol

s3-gateway owns SigV4, S3 request parsing, multipart upload behavior, tags, checksums, and client responses.

Security plane

Identity, policy, keys

Identity, bucket policy, TLS, KMS-style keys, audit events, and credential lifecycle stay isolated from data placement.

Control plane

Bucket state

Bucket configuration, lifecycle, archive, replication, object lock, and website settings are centralized and API-visible.

Metadata plane

Raft / KV / WAL

Metadata is a durable product plane. It may use dedicated PVCs or be colocated with data in per-node PVC deployments.

Data plane

Protected bytes

Object bytes are written as erasure-coded shards or replicated copies, then repaired by background healing when nodes return.

Data Protection Modes

ErasureCoding

Best fit: multi-node clusters where storage efficiency matters and the platform can provide enough failure domains.

How it works: each object is split into dataShards plus parityShards. Data shards contain encoded content. Parity shards allow reconstruction after allowed failures.

Operational rule: metadata is committed only after all required shards are durably written.

Replication

Best fit: small clusters, single-node labs, edge deployments, and cases where simple full-copy behavior is preferred.

How it works: each object is stored as full copies according to the configured replica count.

Operational rule: production settings should still define placement and failure-domain labels so replicas do not silently collapse onto the same risk domain.

Metadata Placement

Colocated with data PVCsPer-node RWO deployments where metadata must land on the same nodes as data.Data and metadata run in the same pod so scheduler placement cannot diverge.
External metadata backendFuture integrations where an approved external metadata system is authoritative.Failure domain defaults to the external storage system and must be validated separately.

The product default is Raft/KV/WAL metadata. The CR should expose placement and capacity choices, not a menu of unimplemented consensus engines.
