Production object storage for OpenShift

S3-compatible storage built from isolated runtime services.

Li9 S3 Storage gives platform teams a product-local S3 service for OpenShift: AWS CLI compatible access, bucket-scoped credentials, policy enforcement, durable metadata, erasure-coded or replicated data placement, lifecycle automation, audit trails, and KMS-style server-side encryption.

Use from an application Review architecture Explore services

What the product delivers

  • S3 endpoint: compatible with standard AWS CLI and SDK endpoint override flows.
  • Bucket onboarding: credentials and endpoint information can be delivered through Kubernetes Secrets.
  • Failure behavior: split brain fails closed instead of accepting divergent writes.
  • Operational visibility: metrics, audit, access logs, health probes, and background repair are first-class concerns.

Primary Use Cases

Application object storage

Use Li9 S3 when application pods need bucket/object APIs without binding the platform to an external cloud account.

OpenShift platform services

Use it for internal logging, backup, data pipeline, and object bucket consumption workflows that need an S3-compatible target.

Controlled data governance

Use product-local policy, lifecycle, archive, replication, audit, encryption, and object lock controls where cloud-only AWS services are not acceptable.

Production Readiness Checklist

1

Placement

Storage and metadata nodes are selected through labels, affinity, tolerations, and explicit failure-domain labels.

2

Protection

Choose erasure coding for multi-node efficiency or replication for simpler small and single-node deployments.

3

Observability

Expose metrics, alerts, dashboards, audit, and access-log delivery before production traffic is moved.

4

Recovery

Validate node loss, pod restart, PVC expansion, degraded writes, and object integrity with real S3 clients.
