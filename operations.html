Operations

Operate performance, observability, scaling, maintenance, and recovery.

Operations documents define how the service behaves after deployment: how to measure throughput, detect degraded writes, expand capacity, drain nodes, recover data, and verify integrity under failure.

Performance

Measure S3 throughput and latency from outside the storage pods. Test small objects, large objects, multipart uploads, sustained writes, and degraded-mode writes.

oc get s3storage -A
oc get pods,pvc,svc,route -n <serviceNamespace>
oc get --raw /apis/custom.metrics.k8s.io/v1beta1 2>/dev/null | head

Establish baseline

Run write, read, list, and multipart tests with the selected protection mode and production-like PVC classes.

Introduce failure

Stop or isolate one storage node while an external client continues writing. Verify write continuity according to the selected durability mode.

Verify healing

Restore the node, watch repair metrics, and compare object checksums after the system returns to healthy state.

Node maintenance and scaling

Before maintenance, check protection mode, failure-domain distribution, metadata quorum, and healing backlog. For RWO per-node PVC mode, storage and colocated metadata move together and are rolled one node at a time.

ChangeExpected operator behaviorAdmin check
Increase PVC size in S3StoragePatch the affected PVCs, verify the requested size, then roll owning workloads where restart is required.oc get pvc -n <serviceNamespace> -o wide

Drain one storage nodeMaintain writes when protection mode still has enough healthy shards or replicas; fail closed otherwise.AWS CLI write loop plus metadata/data health conditions.

Data recovery

Recovery starts from committed metadata. The system should repair missing or stale shards from manifests, reject split-brain writes, and expose status conditions that distinguish metadata, data, and gateway failures.

If metadata quorum is unavailable, the service must fail closed. Accepting writes without consistent metadata creates invisible corruption risk. 
aws --endpoint-url <endpoint> s3 cp ./large-object.bin s3://<bucket>/failure-test/large-object.bin
oc delete pod -n <serviceNamespace> -l app.kubernetes.io/name=li9-s3-storage-node --field-selector spec.nodeName=<node>
aws --endpoint-url <endpoint> s3 cp ./small-object.txt s3://<bucket>/failure-test/small-object.txt
aws --endpoint-url <endpoint> s3api head-object --bucket <bucket> --key failure-test/small-object.txt
