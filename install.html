Installation

Choose one installation path and keep the lifecycle boundary clear.

Li9 S3 Storage can be installed through the OpenShift operator, Helm chart, Linux packages, Windows packages, or a mirrored private-registry flow. Each method owns its own lifecycle and should not be mixed with another method in the same target namespace or host cluster.

Installation Paths

OpenShift Operator

Use this for production OpenShift clusters that need CRDs, cluster-scoped reconciliation, bucket automation, documentation exposure, observability, and OLM upgrades.

Install with OLM

Helm Chart

Use this for Kubernetes environments that do not use OLM or when a GitOps workflow owns chart versions and values directly.

Install with Helm

Linux Packages

Use RPM or DEB packages for VM or bare-metal clusters where systemd owns runtime processes and storage paths are host-managed.

Install on Linux

Windows Packages

Use MSI or ZIP packages for Windows Server clusters where services, firewall rules, and persistent paths are managed outside Kubernetes.

Install on Windows

Private Registry And Airgap

Use this when all images, bundles, charts, and package artifacts must be mirrored into an internal registry before installation.

Prepare a private registry

Minimum Acceptance Test

After any installation path, run a user-level S3 test from outside the storage process boundary.

aws --endpoint-url "${S3_ENDPOINT}" s3 mb s3://li9-install-check
printf 'li9-s3-install-check\n' > /tmp/li9-s3-install-check.txt
aws --endpoint-url "${S3_ENDPOINT}" s3 cp /tmp/li9-s3-install-check.txt s3://li9-install-check/check.txt
aws --endpoint-url "${S3_ENDPOINT}" s3 cp s3://li9-install-check/check.txt -
aws --endpoint-url "${S3_ENDPOINT}" s3 rm s3://li9-install-check/check.txt
aws --endpoint-url "${S3_ENDPOINT}" s3 rb s3://li9-install-check
